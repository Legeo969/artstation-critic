## ADR-001: 使用 Next.js 16 App Router

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 需要选择一个前端框架来构建 Artstation Critic。

**决策**: 使用 Next.js 16 with App Router。

**理由**:
- Server Components 减少客户端包体积
- Server Action 简化 API 调用（无需单独 API 路由）
- 内置路由、布局、加载状态
- 生产就绪，Vercel 一键部署

**后果**:
- 学习曲线（App Router 与 Pages Router 不同）
- 某些浏览器 API 需要在 `'use client'` 中

## ADR-002: 使用 Server Action 调用 AI API

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: AI API 调用需要在服务端还是客户端执行？

**决策**: 使用 Next.js Server Action。

**理由**:
- API Key 不暴露在客户端
- 减少客户端包体积（无需在浏览器加载 OpenAI SDK）
- 代码更简洁（无需单独的 API 路由）

**后果**:
- 需要 `'use client'` 标记客户端组件
- Server Action 不能直接访问浏览器 API

## ADR-003: IndexedDB 本地存储历史记录

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 如何存储用户的点评历史记录？

**决策**: 使用 IndexedDB 本地存储。

**理由**:
- 无需后端服务（降低复杂度）
- 支持大对象存储（base64 图片）
- 浏览器原生支持，无需额外依赖
- 用户数据完全本地，隐私友好

**后果**:
- 数据仅在当前浏览器可用
- IndexedDB 测试在 Node.js 环境下不可用
- 需要处理浏览器兼容性（现代浏览器已支持）

**替代方案考虑**:
- localStorage: 容量限制（~5MB），不适合存图片
- 后端数据库: 需要额外基础设施
- IndexedDB: ✅ 最佳平衡

## ADR-004: DeepSeek 作为默认 AI 提供商

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 选择哪个 AI API 提供商？

**决策**: DeepSeek 作为默认，火山方舟作为备选。

**理由**:
- DeepSeek: 免费额度充足，兼容 OpenAI 格式，改动最小
- 火山方舟: 备选方案，适合国内用户
- 通过 Settings 页面支持动态切换

**后果**:
- 需要处理不同 API 的响应格式差异
- 用户需要自行获取 API Key

**替代方案考虑**:
- OpenAI GPT-4o: 成本高
- Claude 3.5 Sonnet: 无免费额度
- DeepSeek: ✅ 免费 + 兼容

## ADR-005: 中文界面 + 英文品牌

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 界面语言应该用什么？

**决策**: 中文界面，保留英文品牌名 "Artstation Critic"。

**理由**:
- 目标用户主要是中文使用者
- 保留品牌识别度
- 维度名称双语显示（方便对照学习）

**后果**:
- 需要维护中英文翻译
- AI Prompt 需要指定输出语言

## ADR-006: TDD 开发流程

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 采用什么开发流程？

**决策**: 垂直切片 TDD（RED → GREEN → REFACTOR）。

**理由**:
- 每个功能先写测试，确保可测试性
- 小步快跑，快速反馈
- 测试即文档，降低维护成本

**后果**:
- 初期开发速度稍慢
- 长期维护成本降低

## ADR-007: 不实现用户系统（Phase 1）

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: 是否需要用户注册/登录系统？

**决策**: Phase 1 不实现用户系统，使用匿名模式。

**理由**:
- MVP 聚焦核心功能（AI 点评）
- 降低复杂度，快速上线
- 用户数据本地存储，隐私友好

**后果**:
- 数据仅在当前浏览器可用
- 无法跨设备同步

**Phase 2 考虑**:
- 可选的云端同步（用户自愿）
- 不强制登录

## ADR-008: 单图上传优先于批量上传

**日期**: 2026-05-10

**状态**: ✅ 已接受

**背景**: MVP 应该支持单图还是批量上传？

**决策**: MVP 支持单图上传，批量上传放在 Phase 2。

**理由**:
- 降低 MVP 复杂度
- 核心功能验证优先
- 批量上传 UI 更复杂（队列、进度等）

**后果**:
- 用户需要逐个上传
- Phase 2 需要重构上传逻辑

---

## 决策原则

1. **用户价值优先**: 决策应该提升用户体验
2. **技术可行性**: 选择成熟稳定的方案
3. **可逆性**: 优先选择容易回滚的决策
4. **文档化**: 重要决策必须记录原因
